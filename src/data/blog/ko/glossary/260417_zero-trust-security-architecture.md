faqs:
- q: 제로 트러스트 보안이란 무엇인가요?
  a: '''절대 신뢰하지 말고 항상 검증하라''는 원칙 기반의 보안 모델입니다. 네트워크 내외부의 모든 접근 요청을 신뢰하지 않고, 매 순간 엄격한
    인증과 통제를 수행하는 현대적인 보안 프레임워크를 의미합니다.'
- q: 제로 트러스트의 3대 원칙은 무엇인가요?
  a: 명시적 검증(사용자/기기 실시간 확인), 최소 권한 원칙(필요한 권한만 부여), 침해 가정(네트워크가 이미 뚫렸다고 전제하고 대응)의 세
    가지 원칙으로 구성됩니다.
- q: 기존 '성곽과 해자' 방식의 보안과 무엇이 다른가요?
  a: 기존 방식은 외부 침입 차단에만 집중하고 내부는 신뢰하는 경계 보안 중심이었습니다. 반면 제로 트러스트는 경계 자체가 없다고 보고, 위치와
    상관없이 모든 접근 시도를 개별적으로 검증합니다.
- q: 제로 트러스트 보안이 최근 중요해진 이유는 무엇인가요?
  a: 클라우드 도입과 원격·하이브리드 근무 확산으로 물리적 네트워크 경계가 사라졌기 때문입니다. 특히 내부자 실수나 계정 탈취를 통한 내부 이동
    공격을 방어하기 위해 필수적인 대안으로 떠올랐습니다.
- q: 제로 트러스트를 지탱하는 핵심 기술 요소는 무엇인가요?
  a: 사용자 신원을 관리하는 IAM, 다중 인증을 위한 MFA, 특정 앱 접근만 허용하는 ZTNA, 네트워크를 잘게 쪼개는 마이크로 세그먼테이션,
    로그를 분석하는 SIEM 등이 주요 기술입니다.
- q: 기존 VPN과 제로 트러스트(ZTNA)의 결정적인 차이는 무엇인가요?
  a: VPN은 로그인 시 전체 네트워크 접근을 허용하는 경계 기반 방식입니다. ZTNA는 내부 네트워크를 외부에 노출하지 않고, 인증된 사용자에게만
    특정 애플리케이션에 대한 통로를 개별적으로 열어줍니다.
- q: '''침해 가정'' 원칙은 실무에서 어떻게 구현되나요?'
  a: 네트워크가 이미 해커에게 뚫렸을 수도 있다고 가정합니다. 피해 확산을 막기 위해 네트워크를 마이크로 세그먼테이션으로 잘게 쪼개고, 모든 통신
    데이터를 기본적으로 암호화하여 관리합니다.
- q: 제로 트러스트 도입 시 '최소 권한 원칙'은 어떻게 적용되나요?
  a: 특정 업무를 수행하는 데 꼭 필요한 권한만, 필요한 시점에만(Just-in-Time) 부여합니다. 이를 통해 특정 계정이 탈취되더라도 해커가
    접근할 수 있는 범위를 극소화하여 피해를 차단합니다.
- q: 제로 트러스트 보안을 성공적으로 도입한 실제 사례가 있나요?
  a: 구글의 'BeyondCorp'가 대표적입니다. 구글은 이 모델을 통해 직원들이 별도의 VPN 접속 없이도 공용 네트워크에서 사내 리소스를
    안전하게 이용할 수 있는 환경을 구현했습니다.
- q: 제로 트러스트 아키텍처에서 '명시적 검증'이란 구체적으로 무엇인가요?
  a: 사용자의 ID뿐만 아니라 접속 위치, 접속 중인 기기의 보안 상태, 요청한 데이터의 민감도 등 가용한 모든 정보를 실시간으로 조합하여 인증과
    권한 부여를 결정하는 방식입니다.

### 2. 신뢰를 배제한 상시 검증 체계

제로 트러스트는 '아무도 신뢰하지 않는다(Never Trust, Always Verify)'는 전제 아래 모든 접근을 검증하는 보안 모델입니다. 특정 사용자가 사내 네트워크에 접속해 있다는 이유만으로 권한을 부여하던 기존 방식에서 벗어나, 리소스에 접근하려는 모든 주체에 대해 매 순간 엄격한 인증과 통제를 수행하는 것이 핵심이죠.

### 3. 경계 보안의 한계와 패러다임의 전환

과거의 보안 시스템은 이른바 '성곽과 해자(Castle-and-Moat)' 방식으로 운영되었습니다. 외부로부터의 침입만 막아내면 내부 사용자는 안전하다고 판단한 것입니다. 하지만 클라우드 전환이 가속화되고 원격·하이브리드 근무가 일상화되면서 더 이상 물리적인 네트워크 경계를 정의하기가 어려워졌습니다.

특히 내부자 실수나 계정 탈취를 통한 내부 이동(Lateral Movement) 공격에 기존 보안 체계가 무력한 모습을 보이면서, 위치와 상관없이 모든 접근 시도를 개별적으로 검증하는 이 새로운 접근법이 실무적인 대안으로 떠올랐습니다.

### 4. 제로 트러스트를 지탱하는 3대 원칙

- **명시적 검증(Verify Explicitly):** 사용자의 ID는 물론 현재 접속 위치, 기기의 보안 상태, 요청한 데이터의 성격 등 가용한 모든 정보를 조합해 인증과 권한 부여를 실시간으로 결정합니다.
- **최소 권한 원칙(Least Privilege Access):** 업무를 수행하는 데 꼭 필요한 권한만 필요한 시점에(Just-in-Time) 부여합니다. 이를 통해 특정 계정이 탈취되더라도 피해가 확산되는 범위를 최소화할 수 있습니다.
- **침해 가정(Assume Breach):** 네트워크가 이미 해커에게 뚫렸을 수도 있다고 가정합니다. 피해를 국소화하기 위해 네트워크를 잘게 쪼개는 마이크로 세그먼테이션(Micro-segmentation)을 적용하고, 모든 통신 데이터는 기본적으로 암호화하여 관리합니다.

### 5. 기존 방식과의 차이점

| 비교 항목 | 기존 원격 접속 (VPN 등) | 제로 트러스트 (ZTNA) |
| :--- | :--- | :--- |
| **보안 철학** | 경계 기반 (인증 시 내부 신뢰) | 경계 없음 (매 순간 검증) |
| **접근 제어** | 로그인 시 전체 네트워크 접근 허용 | 개별 애플리케이션 및 리소스 단위 제어 |
| **침해 대응** | 내부 침입 후 측면 이동 방어가 어려움 | 세분화된 권한 설정으로 피해 차단 용이 |

### 6. 실무 적용 및 주요 개념

실제 구축 사례로는 구글(Google)의 'BeyondCorp'가 대표적입니다. 구글은 이 모델을 도입해 직원들이 별도의 VPN 접속 없이도 공용 네트워크에서 사내 리소스를 안전하게 이용할 수 있는 환경을 만들었습니다. 

해당 아키텍처를 이해하기 위해 함께 살펴봐야 할 용어들은 다음과 같습니다.

- **IAM (Identity and Access Management):** 사용자의 신원을 확인하고 적절한 자원에 접근할 수 있도록 권한을 관리하는 토대입니다.
- **ZTNA (Zero Trust Network Access):** 내부 네트워크를 외부에 노출하지 않고, 인증된 사용자에게만 특정 앱에 대한 통로를 열어주는 기술입니다.
- **MFA (Multi-Factor Authentication):** 비밀번호 외에 추가적인 인증 수단을 결합하여 신원 확인의 안전성을 높이는 방식입니다.
