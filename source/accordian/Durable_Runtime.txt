Q1: Durable Runtime(내구적 실행)이란 무엇인가요?
A1: 프로세스 생명주기와 실행 상태를 분리하는 기술입니다. 서버 장애나 재시작 시에도 메모리 상태를 잃지 않고, 마지막 성공 지점(Checkpoint)부터 즉시 실행을 재개할 수 있는 '실행의 가상화'를 제공합니다.

Q2: 왜 기존 방식보다 내구적 실행이 중요한가요?
A2: 기존에는 개발자가 DB 저장, 재시도 로직, 상태 머신을 수동 구현해야 했습니다. 내구적 실행은 이를 인프라 계층에서 자동 처리해, 개발자가 복잡한 오류 처리 대신 비즈니스 로직에만 집중하게 돕기 때문입니다.

Q3: 실행 상태를 복구하는 핵심 원리는 무엇인가요?
A3: 이벤트 소싱과 리플레이 모델을 사용합니다. 함수 호출, 타이머 등 결정적 순간을 로그로 기록하고, 프로세스 종료 시 해당 로그를 순차적으로 읽어 이전의 메모리 상태를 그대로 재구성하는 방식입니다.

Q4: 오케스트레이션에서 '멱등성(Idempotency)'이 왜 필요한가요?
A4: 결제나 물류 시스템처럼 여러 단계를 거치는 서비스에서 중복 처리를 막기 위해서입니다. 내구적 실행은 실패한 작업을 정확히 복구하면서도 중복 실행을 방지해 분산 시스템의 신뢰성을 보장합니다.

Q5: 어떤 프로그래밍 언어와 환경에서 사용할 수 있나요?
A5: Microsoft의 Durable Task SDK를 통해 .NET, Python, Java, JavaScript 등 다양한 언어를 지원합니다. 또한 쿠버네티스나 온프레미스 환경에서도 관리형 스케줄러를 통해 클라우드 수준의 운영이 가능합니다.

Q6: 기존의 일반적인 비동기 코드와 구현상 어떤 차이가 있나요?
A6: 문법적으로는 큰 차이가 없지만 인프라가 상태 관리를 추상화한다는 점이 다릅니다. Redis나 큐의 지속성을 직접 관리할 필요 없이 코드 자체에 영속성이 부여되므로 개발 생산성이 비약적으로 높아집니다.

Q7: 내구적 실행 코드를 작성할 때 가장 주의해야 할 점은 무엇인가요?
A7: 반드시 '결정론적(Deterministic)'으로 작성해야 합니다. 동일 입력에 항상 같은 결과가 나와야 리플레이를 통해 상태를 정확히 복구할 수 있습니다. 무작위 값 생성이나 현재 시간 호출 등은 주의가 필요합니다.

Q8: AI 에이전트 구축에 이 기술이 필수적인 이유는 무엇인가요?
A8: AI 에이전트는 수행 시간이 길고 사고 과정 유지가 중요합니다. 내구적 실행은 대기 중 리소스를 절약하고, 중단 시에도 이전 추론 상태를 잃지 않고 재개할 수 있는 최적의 기반 인프라를 제공합니다.

Q9: 관리형 서비스인 'Durable Task Scheduler'의 장점은 무엇인가요?
A9: 상태 관리를 위한 별도 인프라 구성의 번거로움을 덜어줍니다. 스케줄러 엔진을 분리함으로써 시스템의 확장성과 가시성을 확보하고, 다양한 환경에서 일관된 워크플로우 관리를 가능케 합니다.

Q10: 시스템 안정성을 위해 소프트웨어 외에 추가로 고려할 점은 무엇인가요?
A10: 소프트웨어의 내구성은 물리적 인프라 품질과 직결됩니다. 분산 시스템이 성능을 온전히 발휘하려면 안정적인 기업용 전용 회선과 네트워크 인프라가 뒷받침되어 네트워크 연속성이 보장되어야 합니다.
