Q1: eBPF란 구체적으로 어떤 기술인가요?
A1: 리눅스 커널 소스를 수정하지 않고도 커널 내부에서 샌드박스화된 프로그램을 실행할 수 있게 해주는 기술입니다. 네트워크, 보안, 모니터링 기능을 커널 수준에서 동적으로 주입하고 실행할 수 있는 유연성을 제공합니다.

Q2: eBPF가 현대 인프라에서 중요하게 여겨지는 이유는 무엇인가요?
A2: 과거에는 커널 기능을 추가하려면 소스를 직접 수정하거나 위험한 커널 모듈을 로드해야 했습니다. eBPF는 시스템 중단 리스크 없이 안전하고 빠르게 기능을 확장할 수 있는 경로를 제공하여 인프라 혁신을 주도합니다.

Q3: 사용자 정의 코드가 커널을 망가뜨릴 위험은 없나요?
A3: 커널에 로드되기 전 검증기가 무한 루프나 메모리 접근 권한 등을 엄격히 확인합니다. 사용자 코드가 시스템 전체의 안정성을 해치지 않도록 사전에 차단하는 샌드박스 환경을 구축하여 안전한 실행을 보장합니다.

Q4: eBPF 프로그램의 실행 성능은 어느 정도인가요?
A4: 검증을 마친 바이트코드를 JIT 컴파일러가 하드웨어 고유의 머신 코드로 즉시 변환합니다. 인터프리터 방식의 지연 시간을 없애고 커널 네이티브 수준의 빠른 처리 속도를 확보하여 고성능 요구사항을 충족합니다.

Q5: eBPF 프로그램은 시스템의 어느 시점에 실행되나요?
A5: 시스템 호출부터 네트워크 드라이버 계층까지 커널 내 다양한 지점에 부착할 수 있습니다. 특정 이벤트가 발생할 때만 코드가 실행되는 방식이라 리소스 소모를 최소화하면서도 실시간으로 시스템 상황에 대응할 수 있습니다.

Q6: 네트워크 성능 최적화에 사용되는 XDP 기술은 무엇인가요?
A6: XDP는 네트워크 드라이버 계층에서 패킷을 즉시 처리하는 기술입니다. 운영체제의 복잡한 네트워크 스택을 거치기 전에 처리하므로 디도스 방어나 로드 밸런싱의 효율을 극적으로 높일 수 있습니다.

Q7: 쿠버네티스와 같은 클라우드 환경에서 어떻게 활용되나요?
A7: 대표적으로 실리움 프로젝트가 eBPF를 활용합니다. 전통적인 iptables 방식의 성능 오버헤드를 제거하고, 수만 개의 컨테이너 간 복잡한 네트워크 트래픽을 효율적으로 관리하며 세밀한 관측 가능성을 제공합니다.

Q8: 기존 커널 모듈 방식과 비교했을 때 가장 큰 차이점은 무엇인가요?
A8: 커널 모듈은 코드 오류 시 시스템 전체가 멈추는 커널 패닉 위험이 큽니다. 반면 eBPF는 검증기를 통한 샌드박스 실행으로 안전성을 보장하며, 재부팅 없이도 기능을 동적으로 적용하거나 제거할 수 있습니다.

Q9: 보안 강화를 위해 eBPF를 어떻게 실무에 적용할 수 있나요?
A9: 리눅스 보안 모듈과 결합하여 시스템 호출 단계에서 세밀한 보안 정책을 강제할 수 있습니다. 실행 중인 시스템의 런타임 보안 위협을 실시간으로 감지하고 차단하여 인프라의 전반적인 방어 체계를 구축합니다.

Q10: 커널 버전이 다른 여러 서버를 운영할 때 호환성 문제는 없나요?
A10: CO-RE 기술을 활용하면 커널 버전이 서로 다른 환경에서도 재컴파일 없이 프로그램을 실행할 수 있습니다. 이는 BTF를 통해 커널 구조를 파악함으로써 한 번 작성한 프로그램을 다양한 시스템에서 동작하게 돕습니다.
