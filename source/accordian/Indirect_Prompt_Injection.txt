Q1: 인디렉트 프롬프트 인젝션이란 무엇인가요?
A1: 공격자가 AI가 참조하는 웹페이지나 문서 등 외부 데이터에 악의적인 명령을 숨겨두는 공격입니다. 사용자가 AI에게 해당 자료의 처리를 요청하면, AI는 숨겨진 명령을 정상적인 지침으로 오인해 수행하게 됩니다.

Q2: 일반적인 프롬프트 인젝션 공격과는 어떻게 다른가요?
A2: 일반적인 방식은 사용자가 직접 대화창에 명령어를 입력해 AI를 속이는 방식입니다. 반면 인디렉트 방식은 사용자가 아닌 외부 데이터 소스를 매개체로 삼기 때문에 사용자와 보안 시스템이 공격을 인지하기 훨씬 어렵습니다.

Q3: 이 공격이 발생하는 주요 원인은 무엇인가요?
A3: 대규모 언어 모델이 사용자로부터 받은 시스템 지침과 외부에서 읽어온 참조 데이터를 명확하게 구분하지 못하는 구조적 특징 때문입니다. 모델은 모든 텍스트를 동일한 맥락의 명령으로 해석하려는 경향이 있습니다.

Q4: 구체적으로 어떤 피해가 발생할 수 있나요?
A4: AI가 사용자의 세션 토큰이나 민감한 내부 정보를 공격자의 서버로 전송할 수 있습니다. 또한 보안 권한을 오용하여 이메일을 무단으로 발송하거나 시스템 설정을 변경하는 등 심각한 침해 사고로 이어질 수 있습니다.

Q5: 현재 이 위협은 얼마나 심각한 수준인가요?
A5: 마이크로소프트 보안 응답 센터 분석 결과 AI 보안 취약점 중 가장 높은 비중을 차지하고 있습니다. OWASP에서도 2025년 LLM 애플리케이션 보안 리스크 최상위 항목으로 선정하며 그 심각성을 경고하고 있습니다.

Q6: 스포트라이팅(Spotlighting) 기술은 어떻게 공격을 막나요?
A6: 모델이 입력받는 데이터를 신뢰할 수 있는 명령과 검증되지 않은 외부 데이터 계층으로 강제 분리합니다. 특수 구분자를 사용해 물리적 논리 장벽을 세움으로써 데이터 속 명령이 상위 지침을 덮어쓰지 못하도록 유도합니다.

Q7: TaskTracker 기술의 핵심 원리는 무엇인가요?
A7: LLM이 추론을 수행할 때 발생하는 신경망 내부의 활성화 상태를 추적합니다. 정상적인 작업과 공격 명령에 의한 비정상적 작업은 내부 신호 패턴이 서로 다르다는 점을 이용해 실시간으로 공격을 감지하고 차단합니다.

Q8: 데이터 거버넌스가 이 보안 위협과 어떤 관련이 있나요?
A8: 공격자가 명령 주입에 성공하더라도 AI가 접근할 수 있는 권한을 엄격히 제한하면 실제 피해를 최소화할 수 있습니다. 데이터 손실 방지 정책을 통해 AI가 특정 중요 자산에 접근하는 것을 원천적으로 제어해야 합니다.

Q9: 휴먼 인 더 루프(Human-in-the-Loop) 모델이 왜 필요한가요?
A9: 이메일 발송이나 API 호출처럼 외부 시스템에 변화를 주는 작업은 반드시 사용자의 최종 승인을 거치게 하기 위함입니다. 자동화 과정에 인간이 개입함으로써 보안의 결정론적 신뢰를 확보하는 가장 확실한 안전장치입니다.

Q10: 향후 AI 에이전트 시대를 대비한 보안 방향성은 무엇인가요?
A10: AI가 다루는 정보의 흐름을 통제하고 감시하는 다층적 보안 프레임워크를 선제적으로 구축해야 합니다. 보안을 부가적인 옵션이 아닌, 지능형 시스템의 안정성을 지탱하는 가장 본질적인 기반 기술로 재정립해야 합니다.
