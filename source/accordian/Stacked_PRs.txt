Q1: Stacked PRs란 무엇인가요?
A1: 대규모 기능을 논리적 단위로 세분화하여 여러 개의 연관된 풀 리퀘스트를 계층적으로 쌓아 올리는 개발 워크플로우입니다. 큰 작업을 작은 단위로 쪼개어 리뷰 효율을 높이는 것이 핵심입니다.

Q2: 이 방식이 왜 주목받고 있나요?
A2: 대규모 조직에서 코드 리뷰는 병목 현상을 일으키기 쉽습니다. Stacked PRs는 리뷰 크기를 줄여 피드백 루프를 앞당기고, 개발자가 승인 대기 시간 없이 다음 작업을 이어가게 해줍니다.

Q3: Stacked PRs의 주요 특징은 무엇인가요?
A3: 데이터베이스 스키마, 비즈니스 로직, UI 등 기능을 논리적으로 분리해 독립적인 PR을 생성합니다. 각 PR이 이전 단계에 의존하는 계층 구조를 가지며 작업의 맥락을 명확하게 전달합니다.

Q4: 리뷰어 입장에서 어떤 장점이 있나요?
A4: 수천 줄의 코드를 한 번에 검토하는 대신, 수백 줄 이내의 분절된 코드를 논리적 흐름에 따라 순차적으로 확인하므로 인지 부하가 줄어들고 더 꼼꼼한 리뷰가 가능합니다.

Q5: 워크플로우를 유지할 때 가장 큰 어려움은 무엇인가요?
A5: 브랜치 간의 복잡한 의존성 관리입니다. 하위 브랜치가 수정되거나 머지될 때마다 상위 브랜치들을 다시 리베이스하고 동기화해야 하는 수동 조작의 부담이 큽니다.

Q6: 스쿼시 머지를 사용할 때 발생하는 기술적 부채는 무엇인가요?
A6: 하위 PR이 스쿼시 머지되면 커밋 해시가 변경되어 상위 브랜치와의 연결 고리를 잃게 됩니다. 이로 인해 개발자는 수동으로 이력을 재구성해야 하는 번거로운 상황에 직면합니다.

Q7: 기존 기능 브랜치 방식과 비교하면 어떤 차이가 있나요?
A7: 기능 브랜치는 이력 관리가 단순하지만 리뷰 피로도가 높습니다. 반면 Stacked PRs는 리뷰 효율은 극대화되지만 잦은 리베이스와 복잡한 동기화 과정으로 인해 휴먼 에러 위험이 높습니다.

Q8: 실무 도입 시 반드시 고려해야 할 도구가 있나요?
A8: 수동 관리는 위험하므로 Graphite나 jj 같은 전용 도구 사용을 검토해야 합니다. 다만 이런 도구들도 특정 환경에 종속적이거나 설치 제약이 있을 수 있음을 미리 파악해야 합니다.

Q9: 깃으로 Stacked PR 관리할 때 이력이 꼬이지 않게 하려면 어떻게 해야 하나요?
A9: git merge -X ours 같은 강제 명령어를 지양하고, 리베이스 시 git-absorb나 --update-refs 옵션을 활용해 이력을 정교하게 관리해야 합니다. 자동화되지 않은 복잡성은 시스템 안정성을 해칠 수 있습니다.

Q10: 코드 리뷰 속도 올리려고 Stacked PR 쓰는 게 기존 방식보다 진짜 빠른가요?
A10: 리뷰 피드백은 확실히 빠르지만, Git 히스토리를 정리하고 브랜치를 맞추는 데 드는 추가 공수가 상당합니다. 전용 도구 지원 없이 개인의 숙련도에만 의존한다면 오히려 전체 속도가 느려질 수 있습니다.
